# Podmanコンテナ監視・管理スクリプト使用ガイド

## 概要

このドキュメントでは、Podmanコンテナの監視および管理を行うためのスクリプト群の使用方法について説明します。

## ファイル構成

- `container-monitor.sh`: コンテナの状態を監視するスクリプト
- `container-helper.sh`: コンテナ操作を簡略化するための補助スクリプト

## 前提条件

- Podmanがインストールされていること
- Podmanサービスが実行されていること (`podman system service &`)

## 設定

### .envファイルの作成

必要に応じて、環境変数を記述した`.env`ファイルを作成してください。

```bash
# .envファイル例
PODMAN_SOCKET_URL="unix:///var/run/user/$(id -u)/podman/podman.sock"
CONTAINER_LOG_DIR="/var/log/containers"
DEFAULT_TIMEOUT=30
```

### システム設定

1. `container-monitor.sh`に実行権限を付与してください:

```bash
chmod +x container-monitor.sh
```

2. 必要に応じて`container-helper.sh`にも実行権限を付与してください:

```bash
chmod +x container-helper.sh
```

## 使用方法

### コンテナ監視

#### 全コンテナの状態を表示

```bash
./container-monitor.sh --list
```

#### 特定コンテナの詳細情報を取得

```bash
./container-monitor.sh --inspect <コンテナ名またはID>
```

#### リアルタイムでのコンテナ監視

```bash
./container-monitor.sh --watch
```

#### CPU・メモリ使用率を表示

```bash
./container-monitor.sh --stats
```

### コンテナ管理

#### コンテナ起動

```bash
./container-helper.sh start <コンテナ名>
```

#### コンテナ停止

```bash
./container-helper.sh stop <コンテナ名>
```

#### コンテナ再起動

```bash
./container-helper.sh restart <コンテナ名>
```

#### コンテナ削除

```bash
./container-helper.sh remove <コンテナ名>
```

#### ログ表示

```bash
./container-helper.sh logs <コンテナ名>
```

## 便利なオプション

### 監視スクリプト

- `--interval <秒>`: 監視間隔を指定 (デフォルト: 10秒)
- `--health-check`: ヘルスチェックを含めた状態表示
- `--filter-running`: 実行中のコンテナのみ表示
- `--alert-threshold <パーセント>`: CPU/Memory使用率のアラート閾値を設定

### 補助スクリプト

- `--dry-run`: 実際には実行せず、実行予定のコマンドを表示
- `--verbose`: 詳細な出力を表示
- `--timeout <秒>`: タイムアウト時間を設定

## ロギング

コンテナ監視スクリプトは以下の場所にログを出力します:

- `/var/log/containers/monitor.log`: 監視結果のログ
- `/var/log/containers/errors.log`: エラー発生時のログ

ログ出力先は`.env`ファイルの`CONTAINER_LOG_DIR`で変更できます。

## トラブルシューティング

### APIエラーが発生する場合

1. Podmanサービスが実行されていることを確認:

```bash
systemctl --user status podman
```

2. ソケット接続が正しく設定されていることを確認:

```bash
podman info
```

### コンテナが見つからないというエラー

- コンテナ名が完全一致しているか確認してください
- コンテナが実際に存在することを`podman ps -a`で確認してください

## カスタマイズ

### アラート設定

監視スクリプトでアラートをカスタマイズするには、`.env`ファイルに以下の項目を追加してください:

```bash
ALERT_EMAIL="admin@example.com"
SLACK_WEBHOOK_URL="https://hooks.slack.com/services/..."
CPU_THRESHOLD=80
MEMORY_THRESHOLD=85
```

### 通知設定

アラート発生時の通知方法を設定できます:

```bash
ENABLE_EMAIL_ALERTS=true
ENABLE_SLACK_ALERTS=false
ENABLE_LOGGING=true
```

## セキュリティに関する注意

- `.env`ファイルには機密情報が含まれる可能性があるため、適切なアクセス権限を設定してください:

```bash
chmod 600 .env
```

- PodmanのAPIソケットを使用するため、スクリプト実行には適切な権限が必要です
- 生産環境では、APIエンドポイントへのアクセスを制限することを推奨します

# .envファイルの扱い方標準ガイド

## 概要

Podman学習リポジトリ内での`.env`ファイルの扱い方について、標準的なプロセスとベストプラクティスを定義します。

## 目的

- 環境変数の一貫性のある管理
- セキュリティリスクの最小化
- チーム開発における設定の統一

## 基本原則

### 1. ファイル命名規則

- 基本的にプロジェクトルートに`.env`ファイルを配置
- 環境毎に区別が必要な場合は以下のように命名:
  - `.env.development` (開発環境)
  - `.env.staging` (ステージング環境)
  - `.env.production` (本番環境)

### 2. .gitignoreの設定

すべての`.env`ファイルは`.gitignore`に追加することを推奨します。

```
.env
.env.local
.env.development
.env.staging
.env.production
```

例外: サンプル用の`.env.example`ファイルはリポジトリに含めること可

### 3. ファイル構造

```bash
# .env.example (リポジトリに含める)
PODMAN_SOCKET_URL="unix:///var/run/user/$(id -u)/podman/podman.sock"
CONTAINER_LOG_DIR="/var/log/containers"
DEFAULT_TIMEOUT=30
# 空の値で設定例のみ記載
IMAGE_REGISTRY=""
REGISTRY_USER=""
REGISTRY_PASSWORD=""
```

```bash
# .env (ローカル開発環境用、リポジトリには含めない)
PODMAN_SOCKET_URL="unix:///var/run/user/$(id -u)/podman/podman.sock"
CONTAINER_LOG_DIR="/var/log/containers"
DEFAULT_TIMEOUT=30
IMAGE_REGISTRY="docker.io"
REGISTRY_USER="myusername"
REGISTRY_PASSWORD="mypassword"
```

## 実装ガイドライン

### 1. 環境変数の読み込み

シェルスクリプトで`.env`ファイルを読み込む際は、以下の関数を推奨します:

```bash
load_env_file() {
    local env_file="${1:-.env}"
    
    # ファイルが存在しない場合はスキップ
    if [[ ! -f "$env_file" ]]; then
        echo "警告: $env_file が見つかりません。デフォルト設定を使用します。"
        return 0
    fi
    
    # ファイルから環境変数を読み込む
    export $(grep -v '^#' "$env_file" | xargs)
}
```

### 2. デフォルト値の設定

環境変数が設定されていない場合のデフォルト値をスクリプト内で設定します:

```bash
# デフォルト値を設定
PODMAN_SOCKET_URL="${PODMAN_SOCKET_URL:-unix:///var/run/user/$(id -u)/podman/podman.sock}"
CONTAINER_LOG_DIR="${CONTAINER_LOG_DIR:-/tmp/containers}"
DEFAULT_TIMEOUT="${DEFAULT_TIMEOUT:-30}"
```

### 3. セキュリティ対策

- 機密情報（パスワード、トークン等）を含む環境変数は、可能であれば外部シークレット管理システム（HashiCorp Vault、AWS Secrets Managerなど）の使用を検討
- ローカル開発時でも、`.env`ファイルに機密情報を格納する場合は以下のように適切な権限を設定:

```bash
chmod 600 .env
```

### 4. バリデーション

重要な環境変数については、スクリプト開始時にバリデーションを実施:

```bash
validate_required_vars() {
    local required_vars=("PODMAN_SOCKET_URL" "CONTAINER_LOG_DIR")
    
    for var in "${required_vars[@]}"; do
        if [[ -z "${!var}" ]]; then
            echo "エラー: 必須環境変数 $var が設定されていません"
            exit 1
        fi
    done
}
```

## サンプル実装

### 設定読み込みスクリプト

```bash
#!/bin/bash
# config-loader.sh

set -euo pipefail

CONFIG_FILE="${CONFIG_FILE:-.env}"

# .envファイルを読み込む関数
load_env_file() {
    if [[ -f "$CONFIG_FILE" ]]; then
        echo "設定ファイル $CONFIG_FILE を読み込んでいます..."
        export $(grep -v '^#' "$CONFIG_FILE" | xargs)
    else
        echo "警告: 設定ファイル $CONFIG_FILE が見つかりません"
    fi
}

# デフォルト値を設定
set_defaults() {
    PODMAN_SOCKET_URL="${PODMAN_SOCKET_URL:-unix:///var/run/user/$(id -u)/podman/podman.sock}"
    CONTAINER_LOG_DIR="${CONTAINER_LOG_DIR:-/tmp/containers}"
    DEFAULT_TIMEOUT="${DEFAULT_TIMEOUT:-30}"
    LOG_LEVEL="${LOG_LEVEL:-INFO}"
}

# 必須変数を検証
validate_config() {
    if [[ ! -S "${PODMAN_SOCKET_URL#unix://}" ]] && [[ ! "$PODMAN_SOCKET_URL" =~ ^tcp:// ]]; then
        echo "エラー: PODMAN_SOCKET_URL が有効な形式ではありません: $PODMAN_SOCKET_URL"
        exit 1
    fi
}

# 設定を読み込む
load_env_file
set_defaults
validate_config

echo "設定読み込み完了"
echo "  Socket URL: $PODMAN_SOCKET_URL"
echo "  Log Dir: $CONTAINER_LOG_DIR"
echo "  Timeout: $DEFAULT_TIMEOUT"
```

## 保守と運用

### 1. 環境変数のドキュメント化

すべての環境変数について、以下のようなREADMEのセクションを設けて説明を記載:

```
## 環境変数

- `PODMAN_SOCKET_URL`: Podman APIに接続するためのソケットURL（デフォルト: unix:///var/run/user/$(id -u)/podman/podman.sock）
- `CONTAINER_LOG_DIR`: コンテナログの保存先ディレクトリ（デフォルト: /tmp/containers）
- `DEFAULT_TIMEOUT`: API呼び出しのタイムアウト時間（デフォルト: 30）
- `IMAGE_REGISTRY`: イメージレジストリURL（任意）
- `REGISTRY_USER`: レジストリ認証用ユーザー名（任意）
- `REGISTRY_PASSWORD`: レジストリ認証用パスワード（任意）
```

### 2. バージョン互換性

環境変数の変更や廃止がある場合、以下のように対応:

- 新しい変数を追加する際は、古い変数もしばらくサポート
- 廃止予定の変数には、デプレケーション警告を表示
- CHANGELOGで環境変数の変更点を明示

### 3. テスト

`.env`ファイルの読み込みに関するユニットテストを実装:

```bash
# test-config-loading.sh
#!/bin/bash

setup_test_env() {
    cat > test.env << EOF
TEST_VAR=test_value
TEST_NUMBER=123
TEST_BOOLEAN=true
EOF
}

test_load_config() {
    export CONFIG_FILE="test.env"
    source config-loader.sh
    
    assert_equal "$TEST_VAR" "test_value"
    assert_equal "$TEST_NUMBER" "123"
    assert_equal "$TEST_BOOLEAN" "true"
}

teardown_test_env() {
    rm -f test.env
}
```

## 最佳実践

1. **ドキュメント**: `.env.example`ファイルをリポジトリに含め、使用可能な変数とその説明を記載
2. **バリデーション**: 実行前に必要な環境変数が設定されていることを確認
3. **セキュリティ**: 機密情報は外部シークレット管理システムの利用を検討
4. **一貫性**: プロジェクト全体で同じ命名規則と構造を使用
5. **監査**: 変更履歴を追跡し、誰がいつどのような設定を変更したかを把握可能にする
